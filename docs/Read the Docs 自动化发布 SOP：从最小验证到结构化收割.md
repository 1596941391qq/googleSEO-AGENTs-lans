# Read the Docs 自动化发布 SOP：从最小验证到结构化收割

## 阶段一：最小化验证发布 (MVP Launch)
**目标**：验证 Read the Docs (RTD) 在目标关键词下的收录速度与初始排名能力。

### 1. 环境准备
*   **载体选择**：在 RTD 创建一个公开项目，项目名需包含核心行业词（如 `ai-seo-docs`）。
*   **内容标准**：选取 3 个 KD < 20 的长尾词，生成 3 篇具备“事实密度”的 Markdown 文档。
*   **关键配置**：
    *   使用 `index.rst` 或 `index.md` 作为主目录。
    *   在文档末尾植入 **Canonical URL** 指向您的主站（实现权重传递）。

### 2. 执行动作
*   通过 GitHub 仓库同步内容至 RTD。
*   发布后立即在 Google Search Console 手动提交 RTD 的 URL。
*   **监控指标**：记录从发布到 Google 索引的具体小时数。

---

## 阶段二：快慢刀验证 (Velocity & Retention)
**目标**：利用“合法时间差”筛选出具备高曝光潜力的 URL 进行继承。

### 1. 规模化铺设 (快刀)
*   **操作**：在 `/lab/` 路径下，利用 NicheDigger 批量生成 20-50 篇中性实验内容。
*   **逻辑**：这些内容不强求深度，但要求结构清晰，符合 Google 对“可用答案”的初步判断。

### 2. 筛选与继承 (慢刀)
*   **观察期**：7-14 天。
*   **筛选标准**：
    *   **胜出者**：进入 Google 前 50 名或有真实点击的 URL。
    *   **淘汰者**：无曝光、无索引的 URL。
*   **继承动作**：
    *   将胜出 URL 的内容升级为“数据支撑型深度内容”。
    *   通过 **301 重定向** 或 **Canonical 迁移**，将权重引导至 `/guide/` 或 `/compare/` 路径下的永久页面。

---

## 阶段三：内链结构验证 (Link Sculpting)
**目标**：通过“单向漏斗”模型，人为制造“主题中心页”的必然性。

### 1. 实施“同词簇放大”模型
*   **结构布局**：
    *   创建 5 篇 `/lab/` 页面，分别讨论某个大主题的子维度。
    *   **强制内链**：这 5 篇页面全部单向链接至 1 篇 `/guide/` 核心枢纽页。
    *   **Anchor Text**：使用“精确匹配”与“语义描述” 1:1 混合。

### 2. 实施“Facet 伪导航”
*   **结构布局**：
    *   在核心枢纽页下方，创建多个 `facet` 页面（如：`for-beginners`, `enterprise-edition`）。
    *   **规则**：只允许 `facet → hub`，严禁 `facet` 之间互链。
*   **验证点**：观察 Google 是否将该枢纽页识别为该主题的“权威节点”。

---

## 阶段四：自动化收割与交付
*   **转化出口**：在所有胜出的高权重页面中，植入指向用户官网的 **CTA 按钮** 或 **表单**。
*   **系统自治**：NicheDigger Agent 根据排名反馈，自动决定哪些内容需要“升级继承”，哪些需要“无情删除”。

---

## 附录：RTD 发布避坑指南
*   **禁止商业化过度**：RTD 是技术文档平台，初期内容应保持“中性、教育性”，转化链接应放在文章中后部。
*   **保持更新频率**：Google 对 RTD 的信任源于其“持续维护”的假象，建议 Agent 每周对胜出页面进行微小的数据更新。
